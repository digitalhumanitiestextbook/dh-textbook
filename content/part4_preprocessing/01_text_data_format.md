## 텍스트 데이터 형식

> 💡 **인문학적 질문**
>
> "우리가 마주할 텍스트 데이터는 어떤 모습일까요?"

인문학자가 마주하는 데이터는 단순히 숫자들의 나열이 아닌, 복잡하고 다층적인 의미를 담은 텍스트인 경우가 많습니다. 이러한 텍스트 데이터는 다양한 형식으로 존재하며, 각 형식의 특징과 장단점을 이해하는 것은 "어떤 연구 질문에 어떤 데이터 형식이 적합한가"를 판단하는 첫 단추를 꿰는 것과 같습니다.

## 1.1 Plain Text (.txt): 가장 순수한 텍스트
가장 순수하고 기본적인 텍스트 형식입니다. 소설, 시, 대본 등 일체의 구조 정보 없이 오직 글자만 담겨 있습니다. 우리가 메모장에서 흔히 만드는 파일이 바로 플레인 텍스트입니다. 윤동주 시인의 시나 『홍길동전』과 같은 고전 소설 원문이 이 형식으로 공유되곤 합니다.

* **장점:** 거의 모든 컴퓨터 환경에서 쉽게 열고 읽을 수 있으며, 용량이 작습니다.
* **한계:** 텍스트의 구조(예: 이 부분이 제목인지, 저자인지, 본문인지)에 대한 정보가 전혀 없어, 컴퓨터가 내용의 의미를 파악하기 매우 어렵습니다. "윤동주 시인의 시 중에서 "하늘"이라는 단어가 들어간 시의 제목만 모두 추출해줘"와 같은 작업을 하려면, 각 시의 제목과 본문을 구분하는 복잡한 규칙을 사람이 직접 만들어야 합니다.

## 1.2 CSV/TSV: 반듯한 표(Table)로 정리된 데이터
"Comma-Separated Values" 또는 "Tab-Separated Values"의 약자로, 엑셀 시트처럼 행(row)과 열(column)로 이루어진 표 형식의 데이터를 표현합니다. 각 데이터 값이 쉼표나 탭으로 구분되어 있어 구조가 명확합니다. 정부의 공공데이터포털(data.go.kr)에서 제공하는 많은 데이터가 이 형식이며, 인문데이터 분석에서도 인물 목록, 연표, 어휘 목록 등을 정리하는 데 매우 유용합니다.

* **장점:** 구조가 단순하고 명확하여 기계가 읽기 쉽고, 대부분의 데이터 분석 도구(엑셀, R, Python Pandas 등)에서 곧바로 활용할 수 있습니다.
* **한계:** 계층 구조나 복잡한 관계를 표현하기 어렵습니다. 예를 들어, 한 명의 인물이 여러 개의 관직을 가졌거나, 여러 개의 다른 이름(아명, 호 등)을 가진 경우를 표 안에 깔끔하게 표현하기는 까다롭습니다.

**예시 1: 역사 인물 정보**

* 질문: "18세기 영조 시대에 활동한 문신들의 평균 관직 생활 기간은 얼마였을까?"
```csv
인물,출생,사망,주요관직
이순신,1545,1598,삼도수군통제사
세종대왕,1397,1450,왕
정약용,1762,1836,형조참의
```

**예시 2: 1990년대 인기 가요 목록**

* 질문: "1995년에 가장 많은 인기곡을 발표한 가수는 누구일까?"

```tsv
가수	노래제목	발표년도
서태지와 아이들	하여가	1993
김건모	잘못된 만남	1995
H.O.T.	캔디	1996
```

## 1.3 JSON: 웹과 소통하는 유연한 데이터
"JavaScript Object Notation"의 약자로, 웹 서비스들이 서로 데이터를 주고받을 때(API 통신) 널리 사용되는 형식입니다. 2장에서 배운 파이썬의 딕셔너리(`{Key: Value}`)와 리스트(`[]`)가 중첩된 형태로 정보를 표현하여, 단순한 표보다 훨씬 유연하고 복잡한 구조를 담을 수 있습니다.

* **장점:** 계층적인 데이터를 표현하는 데 매우 효과적입니다. 웹 기반의 데이터를 수집할 때(예: 영화진흥위원회 API, 국립국어원 API) 대부분 JSON 형식을 만나게 되므로 반드시 알아두어야 합니다.
* **한계:** 데이터 구조가 복잡해지면 사람이 한눈에 파악하기 다소 어려울 수 있습니다.

**예시 1: 영화 정보** 

* 질문: "봉준호 감독의 영화 중 2010년대에 개봉한 영화들의 평균 누적 관객수는 얼마일까?"

```json
{
  "감독명": "봉준호",
  "필모그래피": [
    {
      "영화명": "기생충",
      "개봉년도": 2019,
      "관객수": 10313201
    },
    {
      "영화명": "옥자",
      "개봉년도": 2017,
      "관객수": 322673
    }
  ]
}
```

**예시 2: 서울시 공공자전거 "따릉이" 대여소 정보**

* 질문: "마포구에 있는 따릉이 대여소는 총 몇 개인가?"

```json
{
  "요청결과": "성공",
  "대여소목록": [
    {
      "대여소ID": "ST-419",
      "대여소명": "DMC역 8번출구 앞",
      "소재지": "마포구",
      "거치대수": 20
    },
    {
      "대여소ID": "ST-10",
      "대여소명": "마포구청 앞",
      "소재지": "마포구",
      "거치대수": 15
    }
  ]
}
```

## 1.4 XML/TEI: 인문학의 지혜가 담긴 데이터
XML("eXtensible Markup Language")은 사용자가 직접 태그(`<tag>`)를 정의하여 데이터의 구조와 의미를 함께 표현할 수 있는 매우 유연한 형식입니다. 

특히 "**TEI(Text Encoding Initiative)**"는 XML을 기반으로 고문서, 문학 텍스트, 역사 기록 등 복잡하고 비정형적인 인문학 자료를 기계가 읽을 수 있는 형태로 구조화하기 위해 전 세계 인문학자들이 함께 만든 국제 표준 규약입니다. 이는 단순히 데이터를 저장하는 것을 넘어, 텍스트에 대한 연구자의 "해석"을 코드화하는 행위입니다. 

예를 들어, 어떤 부분을 시의 "연(stanza)"으로 볼 것인지, 소설의 "인용문(quote)"으로 볼 것인지를 태그로 명시하는 것 자체가 하나의 학술적 주장인 셈입니다. 한국에서도 『조선왕조실록』 CD-ROM 개발이나 『고려대장경』(팔만대장경)의 디지털화 사업에서 이러한 구조화된 텍스트의 중요성이 일찍부터 강조되었습니다.

* **장점:** 인간의 언어가 가진 복잡성, 중의성, 계층성을 매우 정교하게 표현할 수 있습니다. 텍스트 원문(facsimile)과 판독문(diplomatic), 교정문(critical) 등 다양한 층위의 정보를 함께 담을 수 있어 문헌학 연구에 필수적입니다.
* **한계:** 구조가 복잡하고, 데이터를 제대로 구축하려면 해당 분야의 전문 지식과 TEI 규약에 대한 학습이 필요합니다.

**예시 1: 훈민정음 언해본**

* 질문: "『훈민정음』에서 세종의 발언과 집현전 학사들의 설명을 구분하여 분석할 수 있는가?"

```xml
<sp who="#세종">
    <speaker>세종</speaker>
    <p>나라의 말이 중국과 달라 문자와 서로 통하지 아니하므로...</p>
</sp>
```

**예시 2: 윤동주의 시 "서시"**

* 질문: "윤동주의 시에서 "밤"이라는 시어는 어떤 맥락(행, 연)에서 주로 등장하는가?")

```xml
<lg type="stanza" n="1">
    <l n="1">죽는 날까지 하늘을 우러러</l>
    <l n="2">한 점 부끄럼이 없기를,</l>
    <l n="3">잎새에 이는 바람에도</l>
    <l n="4">나는 괴로워했다.</l>
</lg>
```

## 1.5 RDF: 관계를 위한 데이터, 지식그래프의 씨앗
"Resource Description Framework"의 약자로, 세상의 모든 지식을 "주어-서술어-목적어"라는 최소 단위(이를 "**트리플(Triple)**"이라 부릅니다)로 표현하는 데이터 형식입니다. 

예를 들어 "세종대왕은 훈민정음을 창제했다"는 정보를 `<세종대왕> <창제했다> <훈민정음>` 과 같은 형태로 저장합니다. 이렇게 만들어진 관계의 조각들이 모여 거대한 "지식그래프(Knowledge Graph)"를 이룹니다.

* **특징:** 개별 데이터보다는 데이터 간의 "관계"를 표현하는 데 특화되어 있습니다. 이후에 배울 네트워크 분석이나 시맨틱 웹, 링크드 데이터(Linked Data) 기술의 핵심적인 기반이 됩니다. 한국학 분야에서도 인물, 사건, 지명, 문헌 간의 관계를 RDF로 구축하여 "한국사 지식그래프"를 만들려는 시도가 활발히 이루어지고 있습니다.
